ValueOS UX Redesign: Principal Product Designer Deliverable
Phase 1: Product Understanding
1.1 Top 3 Personas
Primary: The Strategist (Pre-Sales Consultant / Value Engineer)
* Goal: Rapidly build credible, data-driven value hypotheses that make the solution the only logical choice
* Anxieties:
   * "Will this number embarrass me in front of the CFO?"
   * "I don't have time to research every prospect manually"
   * "The model looks generic—it won't resonate"
* Success metric: Time-to-first-draft < 15 minutes; hypothesis acceptance rate by prospects
Secondary: The Closer (Account Executive / Sales Leader)
* Goal: Convert the value hypothesis into a commercial win with executive-ready narratives
* Anxieties:
   * "I can't show this spreadsheet to a CFO"
   * "What if I accidentally change something during the presentation?"
   * "I need to answer objections on the fly"
* Success metric: Deal velocity; win rate; average contract value
Admin: The Grower (Customer Success Manager)
* Goal: Prove delivered value and identify expansion opportunities
* Anxieties:
   * "Did we actually deliver what we promised?"
   * "I need proof before the QBR, not excuses"
   * "Where's the upsell opportunity?"
* Success metric: Net Revenue Retention; QBR preparation time; expansion pipeline

Workflow 1: Discovery → First Value Model
1. User enters company name/URL
2. Agent researches (10-K, LinkedIn, news)
3. Agent proposes value drivers with confidence levels
4. User reviews/adjusts assumptions
5. Agent generates first-draft model
6. User refines collaboratively
Workflow 2: Model → Executive Presentation
1. User requests "presentation mode"
2. Agent transforms model into narrative arc
3. User selects persona (CFO, CTO, COO)
4. Agent generates talking points
5. User presents with live "what-if" toggles
6. Changes sync back to master model
Workflow 3: Commitment → Value Realization
1. User formalizes target KPIs
2. System creates ValueCommitment artifact
3. Post-sale: Grower inherits commitment
4. Agent tracks actuals via integrations
5. Agent generates QBR-ready proof
6. Agent identifies expansion whitespace
1.4 Critical Edge Cases
Edge Case
	Current Handling
	Required Handling
	Agent can't find data (403, no 10-K)
	Generic error toast
	Agent offers alternatives: "Upload PDF" or "Use industry benchmarks"
	User changes tenant mid-flow
	Implicit, anxiety-inducing
	Explicit confirmation modal with data scope warning
	Conflicting assumptions
	Silent override
	Agent flags conflict: "You said 2,000 employees earlier, but the 10-K shows 1,800. Which should I use?"
	Approval required (cost/compliance)
	Not surfaced
	Approval card with clear stakes: "This will sync to Salesforce. Proceed?"
	Session timeout
	Lost work
	Auto-save + "Resume where you left off" on return
	1.5 Key Constraints
Constraint
	Design Implication
	Multi-tenant
	Persistent tenant badge; explicit switcher; no cross-tenant data leakage
	RBAC
	Role-aware UI morphing; hidden actions for insufficient permissions
	Audit requirements
	Every agent action logged; "View audit" one-click from any output
	Billing/cost controls
	Pre-action cost estimates; usage dashboard; alerts before overages
	Compliance (SOC2, etc.)
	Data residency indicators; export restrictions by tenant policy

	2.3 Steering & Correction Mechanisms
Undo/Redo:
* Every agent action creates a checkpoint
* User can click any message in history to "restore to this point"
* Canvas updates to match historical state
Assumption Override:
* Inline editing in Plan cards
* Natural language: "Actually, use 1,500 employees instead"
* Agent acknowledges and recalculates
Error Recovery:
* Agent offers 2-3 alternatives, not just "retry"
* User can provide missing data inline
* "Help me understand" escalation path

Error Pattern Principles:
1. Human language: "I got stuck" not "Error 403"
2. Explain why: Brief, non-technical reason
3. Offer alternatives: 2-3 actionable paths forward
4. Progressive disclosure: Technical details hidden by default
5. Preserve context: Don't lose user's work

* Error boundary components at route and panel level
* Centralized error classification (tool_failure, permission_denied, config_missing, rate_limit)
* Emit: error_displayed, error_recovery_action
