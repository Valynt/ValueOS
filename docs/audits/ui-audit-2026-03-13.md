# UI Audit — ValueOS (ValyntApp)

_Date:_ 2026-03-13  
_Scope:_ `apps/ValyntApp/src` with focus on routing/workflows/layout/components/functions.

## Executive summary

The UI codebase has strong foundational structure (feature folders, route guards, lazy-loading, Suspense, provider composition), but currently shows architectural drift from parallel navigation systems, mixed legacy/new layouts, and convention misalignment (notably default exports and directory casing duplicates). The highest-impact risks are route duplication/ambiguity and dead-path UX divergence.

## Audit method

- Read route composition and guards in `AppRoutes.tsx` and `app/routes/index.tsx`.
- Reviewed key layouts (`MainLayout`, `AppLayout`, `AppShell`, `MarketingLayout`) and workflow surfaces (`features/workflow`, `components/Workflow`).
- Ran lightweight static checks for route volume, casing collisions, and default export prevalence.

## Findings

## 1) Workflows (user flows and route orchestration)

### ✅ Strengths

- Route guards are present and explicit (`ProtectedRoute`, `PermissionRoute`, `TenantGate`, `OnboardingGate`), which supports staged onboarding and permission-gated surfaces.
- Lazy loading is consistently used for major views to reduce first-load overhead.

### ⚠️ Issues

1. **Parallel route systems indicate architecture drift**
   - `src/AppRoutes.tsx` is the active production route tree.
   - `src/app/routes/index.tsx` defines a second, broad route tree with overlapping concerns.
   - This increases maintenance cost, test ambiguity, and onboarding confusion.

2. **Route security stance is inconsistent between route trees**
   - `app/routes/index.tsx` labels `/workspace` and `/app` as “unprotected for dev”.
   - In `AppRoutes.tsx`, app core is protected and tenant-gated.
   - If both files are referenced by different entry points/tests, behavior can diverge in ways hard to detect.

3. **Potential route duplication/ambiguity in `app/routes/index.tsx`**
   - The path `"/app/settings"` appears under nested settings and then again mapped directly to `ValueOSHome`, which is likely unintended and can mask regressions.

## 2) Layouts and navigation architecture

### ✅ Strengths

- Layout intent is clear (marketing shell vs app shell vs main authenticated shell).
- Accessibility basics are present in multiple layouts (skip links, aria labels).

### ⚠️ Issues

1. **Multiple overlapping layout systems**
   - `MainLayout`, `AppLayout`, and `AppShell` all present navigation shells with differing UX and information architecture.
   - This suggests partial migration and can produce inconsistent interaction patterns.

2. **Directory casing collisions in components**
   - Both `components/common` and `components/Common` exist.
   - Both `components/layout` and `components/Layout` exist.
   - This is fragile across case-sensitive/case-insensitive environments and raises import hygiene risk.

## 3) Components

### ✅ Strengths

- Workflow UI components include accessibility and clear state modeling (`pending/active/completed/failed/skipped`).
- Feature foldering in `features/workflow` and `features/workspace` is generally coherent.

### ⚠️ Issues

1. **Convention drift: heavy use of default exports**
   - Root guidance requires named exports only.
   - Current frontend includes widespread default exports (160 occurrences in `apps/ValyntApp/src`).

2. **Potential compile/runtime defect in `AppShell`**
   - `useId()` is used but not imported from React in `components/layout/AppShell.tsx`.
   - This should fail TypeScript or runtime depending on build path and dead-code status.

## 4) Functions / hooks / service composition

### ✅ Strengths

- `useWorkflow` exposes a cohesive hook API with local state transitions and explicit error handling.
- Query cache is proactively cleared on tenant switch in `AppRoutes.tsx`.

### ⚠️ Issues

1. **Workflow docs are generic and low-signal**
   - `features/workflow/README.md` is template-like and doesn’t document lifecycle states, data contracts, or integration boundaries.

2. **Potential dead code accumulation**
   - Presence of legacy view paths and alternate route configs suggests non-trivial dead or partially orphaned UI surfaces.

## Priority recommendations

## P0 (stability / correctness)

1. **Consolidate to a single route source of truth**
   - Keep `src/AppRoutes.tsx` (or successor), deprecate duplicate route trees, and add tests that assert only one exported router is used at app entry.

2. **Remove or quarantine dev-unprotected route definitions**
   - Ensure production route artifacts cannot include “unprotected for dev” pathways.

3. **Fix `AppShell` React hook import defect (`useId`)**
   - Either import correctly and verify usage, or remove/retire `AppShell` if deprecated.

## P1 (maintainability)

1. **Normalize directory casing**
   - Merge `Common/common` and `Layout/layout` with codemod import rewrites.

2. **Enforce named-export convention**
   - Add ESLint rule(s) and rollout plan to reduce default exports incrementally.

3. **Document workflow contracts**
   - Replace placeholder `features/workflow/README.md` with state machine description, API schema, and UI event map.

## P2 (quality / platform readiness)

1. Add route topology snapshot tests (guard + redirect matrix).
2. Add UI architecture ADR documenting active layout/router and migration strategy.
3. Add automated checks for case-collision paths in CI.
