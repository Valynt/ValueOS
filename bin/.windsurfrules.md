# ValueOS Workspace Rules

## Architecture Standards

- All agents must inherit from `BaseAgent` in `src/lib/agent-fabric/agents/BaseAgent.ts`.
- Every agent-to-agent communication must be logged via `SecureMessageBus` in `src/lib/agent-fabric/SecureMessageBus.ts`.
- Data flow between services must prioritize the "Causal Truth" schema defined in `blueprint/value_fabric/`.
- Workflows must follow DAG patterns (cycles FORBIDDEN) as per `docs/adr/0001-orchestration-layer.md`.

## Security & Compliance

- No LLM invocation should bypass the budget tracker in `src/lib/llm-gating/BudgetTracker.ts`.
- Sensitive PII data must be masked using `src/lib/piiFilter.ts` before being sent to any LLM service.
- All database migrations must be reviewed against `docs/database-migration-policy.md`.
- GDPR and SOC2 compliance requirements are documented in `docs/compliance/`.

## Agent Development

- Each agent must be a single class in `[AgentName]Agent.ts` extending `BaseAgent`.
- Use handlebars syntax for prompts (string concatenation FORBIDDEN).
- All LLM calls must go through `this.secureInvoke()` ONLY.
- Mock `LLMGateway` and `MemorySystem` in tests.
- Agent specifications are defined in `.github/agent-prompts/master-orchestrator.md`.

## Dev Environment

- Infrastructure changes must be validated against `infra/compose/compose.dev.yml`.
- Port mappings must be updated in `config/ports.json` whenever a new service is added.
- Environment variables for dev are managed in `infra/environments/dev/`.
- DevContainer scripts are located in `.devcontainer/scripts/`.

## Codemap Integration

- Use the `@codemap` feature to trace side effects before refactoring any service in `src/services/`.
- The `valueos-codemap` extension (`.vscode-extension/valueos-codemap/`) provides:
  - Live dependency graphs
  - Blast radius analysis
  - Hot path detection via Prometheus metrics
- Generate Codemaps for agentic flows to verify implementation matches architecture docs.

## Data Flow Validation

- All data persistence in the Value Fabric must include a provenance trace as defined in `src/lib/agent-fabric/MemorySystem.ts`.
- Trace data flows from frontend through `blueprint/value_fabric/postgres_schema.sql`.
- Use Codemaps to identify missing provenance logic in data pipelines.

## Testing Requirements

- 100% test coverage required for agent modules.
- Services must NOT hold state between requests.
- Multi-table writes must use SQL transactions (via RPC).
- Bypass RLS with `service_role` ONLY for: AuthService, TenantProvisioning, CronJobs.

## Value Fabric Migration Rules

**CRITICAL**: All migrations in `blueprint/value_fabric/` and `supabase/migrations/` must be visualized on a Codemap before being applied.

### Pre-Migration Checklist

1. Generate a `@{migration-impact-map}` showing affected tables and relationships
2. Verify migration follows `docs/database-migration-policy.md`
3. Check for breaking changes to `postgres_schema.sql`
4. Ensure rollback script exists for destructive operations
5. Validate RLS policies are preserved or updated

### Migration Visualization Rule

```
@{migration-impact-map}: Trace migration SQL → affected tables → dependent services
Required before: Any ALTER TABLE, DROP, or schema modification
```

### Validation Prompt

> "Before applying this migration, generate @{migration-impact-map} to show:
>
> 1. Which tables are affected
> 2. Which services depend on these tables
> 3. Whether RLS policies need updating
> 4. Estimated downtime or data migration requirements
>    Block the migration if any Value Fabric provenance columns are removed."
