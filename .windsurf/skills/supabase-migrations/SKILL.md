Instruction Set: supabase-migrations1. GoalTo audit, clean, and prepare Supabase migrations for production release by distinguishing between functional SQL (Real) and boilerplate/empty files (Stubs), while ensuring security compliance (RLS).2. Core OperationsA. Stub Detection & CleanupIdentify migrations that add no value to the schema to prevent "migration bloat."Identify Stubs: Scan files in /supabase/migrations/*.sql that are under 100 bytes or contain only -- empty migration or BEGIN; END;.Action: If a file is a stub, recommend deletion or "Repair" if it has already been recorded in the remote history.B. Security & RLS ValidationEvery table created in a migration must have an associated security policy.Rule: Every CREATE TABLE statement must be followed by ALTER TABLE <name> ENABLE ROW LEVEL SECURITY;.Rule: Look for a corresponding CREATE POLICY statement for each table. If missing, flag as "High Risk."C. Idempotency CheckEnsure migrations can be re-run in CI/CD without crashing.Requirement: Use IF NOT EXISTS for tables, indexes, and extensions.Requirement: Use OR REPLACE for functions and views.3. Workflow Commands (Reference)When this skill is active, you should prioritize these CLI patterns:To Baseline ValueOS:supabase db diff --local > supabase/migrations/00000000000000_baseline.sqlTo Verify Parity:supabase db reset (Crucial: If this fails, your migrations are missing dependencies).4.
